# 🐾 Open Pet Registry

**A free, open-source, privacy-first lost-pet microchip registry. Its only mission is
reuniting lost pets with the people who love them.**

This is public infrastructure, not a business. Core reunification features are **free
forever**, the software is open source, and owner data is **encrypted and never sold**.
Possessing a microchip number never exposes an owner's personal information.

> Why this exists: prior registries have failed by trapping their data on a single
> company's servers, leaving pets un-findable when that company disappeared. This project is
> designed to outlive any single maintainer, company, or cloud provider. See
> [`docs/spec.md`](docs/spec.md) for the full vision, and the research in
> [`docs/phase1results.md`](docs/phase1results.md) / [`docs/phase2.md`](docs/phase2.md).

## What it does (MVP)

- **Register a pet** — species, breed, photo, microchip, emergency contacts.
- **Passwordless sign-in** — magic links; no passwords to leak.
- **Secure microchip lookup** — a finder enters a chip number, the **owner is notified**,
  and the finder learns *nothing* about who owns the pet.
- **QR pages & printable posters** — put a QR on a collar tag; flip on **Lost Mode** to turn
  the public page into a LOST alert with a printable poster.
- **Encryption at rest** — owner/contact/finder PII is sealed with per-record keys; a
  database leak yields ciphertext, not people.

## Quick start

Requires **Node 20+**.

```bash
npm install
npm run setup     # generate keys + migrate the DB + seed demo data
npm run dev       # http://localhost:3000
```

`npm run setup` prints demo microchip numbers and public pet URLs. Try a lookup at
`/lookup`, or sign in at `/auth/request` — in development, magic-link and notification emails
are printed to the server console and written to `./var/dev-outbox.jsonl`.

## How it's built

Next.js (App Router) + TypeScript · Drizzle ORM over **Turso / libSQL** (SQLite-compatible,
Postgres-swappable) · self-hosted magic-link auth · app-level **envelope encryption**
(AES-256-GCM, per-record keys, master key kept out of the database).

Strict layered architecture, enforced in CI:

```
app → services → { domain · db · providers · crypto · security }
```

Every external dependency (email, SMS, storage, AI, poster) sits behind a provider
interface, so nothing is locked to one vendor. See [`docs/spec.md`](docs/spec.md) and the
[ADRs](docs/adr/).

## Commands

| Command | What it does |
|---|---|
| `npm run dev` | Start the dev server |
| `npm run setup` | Keygen → migrate → seed demo data |
| `npm run keygen` | Generate encryption keys into `.env` (never overwrites existing keys) |
| `npm run db:generate` | Generate a new SQL migration from the schema |
| `npm run db:migrate` | Apply migrations |
| `npm run seed` | Insert demo owners/pets/chips |
| `npm test` | Unit + integration tests |
| `npm run test:e2e` | Playwright end-to-end tests |
| `npm run typecheck` / `npm run lint` | Type + lint (incl. architecture boundaries) |
| `npm run build` / `npm start` | Production build / serve |

## Configuration

All settings live in `.env` (see [`.env.example`](.env.example)). Keys are generated by
`npm run keygen` and **must never be committed**. Email defaults to a console/dev transport;
set `EMAIL_PROVIDER=smtp` with any SMTP host for production. Database defaults to a local
SQLite file; point `DATABASE_URL` at a Turso URL for hosted deployment.

## Security

Owner PII is encrypted with envelope encryption; the master key lives outside the database.
See [`docs/THREAT-MODEL.md`](docs/THREAT-MODEL.md), [`docs/KEY-MANAGEMENT.md`](docs/KEY-MANAGEMENT.md),
and [`SECURITY.md`](SECURITY.md) to report a vulnerability.
